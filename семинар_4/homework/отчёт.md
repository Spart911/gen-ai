**Эмбеддер:** `paraphrase-multilingual-MiniLM-L12-v2`

## 3. Результаты

### hit-rate@5 (hybrid)

| Метрика | fixed (2000 симв.) | recursive (400 / overlap 80) |
|---------|-------------------:|-----------------------------:|
| **hit-rate@5** | **1.00** (15/15) | **1.00** (15/15) |
| Чанков в индексе | 85 | 594 |
| Средний ранг gold-дока | **1.17** | 1.72 |

### Baseline: dense-only

| | fixed | recursive |
|---|------:|----------:|
| hit-rate@5 | 0.97 | 0.97 |

Обе стратегии чанкинга при hybrid-поиске закрывают весь gold-сет. Различия — в размере индекса и позиции gold-документов в top-5.

## 4. Анализ ошибок и различий

На hit-rate@5 обе стратегии **не проигрывают друг другу**, но в ретриве видны разные паттерны — ниже три показательных случая.

### Q1 — multi-hop (id 1) · dense-only «ломается», hybrid спасает

**Вопрос:** бот по базе знаний + память диалога.  
**Gold:** `doc_01` + `doc_03`.

| Retrieval | fixed | recursive |
|-----------|-------|-----------|
| Hybrid | ✓ 1.00 — оба документа в top-5 | ✓ 1.00 |
| Dense-only | ✗ 0.50 — только `doc_01` | ✗ 0.50 — только `doc_03` |

**Fixed + dense-only:** top-5 = четыре чанка `doc_01__*`, один `doc_02`; `doc_03` не попал.  
**Recursive + dense-only:** все пять чанков — `doc_03__*`, `doc_01` не попал.

Dense «залипает» на одном семантически близком документе. BM25 по словам `ChromaDB`, `база знаний`, `messages` добавляет второй источник — hybrid даёт 1.00 независимо от чанкинга.

**Hybrid, fixed:** `doc_03__4` (rank 1), `doc_01__0` (rank 2).  
**Hybrid, recursive:** `doc_03__24` (rank 1), `doc_01__17` (rank 2) — та же пара документов, более мелкие чанки.

---

### Q6 — multi-hop (id 6) · оба hit=1.00, но разное ранжирование

**Вопрос:** контекстное окно ↔ токены API ↔ VRAM.  
**Gold:** `doc_01` + `doc_04`.

| | fixed | recursive |
|---|-------|-----------|
| hit@5 | 1.00 | 1.00 |
| Ранг `doc_04` (VRAM) | **1** | **5** |
| Ранг `doc_01` (API) | 2 | 1–4 (4 чанка в top-5) |

**Fixed** ставит `doc_04__0` (тред про железо под LLM) на первое место — в одном чанке 2000 симв. сразу много слов про контекст и VRAM.  
**Recursive** заполняет top-4 чанками `doc_01` (треды про API/контекст); `doc_04__5` проскакивает лишь на 5-й позиции. Для генерации ответа LLM это риск: VRAM-контекст может оказаться на «хвосте» контекстного окна.

---

### Q11 — multi-hop (id 11) · recursive «на грани» по второму документу

**Вопрос:** pandas без циклов + подсчёт подряд идущих значений.  
**Gold:** `doc_08` + `doc_09`.

| | fixed | recursive |
|---|-------|-----------|
| hit@5 | 1.00 | 1.00 |
| Ранг `doc_08` | **1** | **1** |
| Ранг `doc_09` | **2** | **5** |

**Fixed:** `doc_08__0` (rank 1), `doc_09__0` (rank 2) — оба gold-документа в начале выдачи.  
**Recursive:** top-4 — чанки `doc_08__*` (groupby / split-apply-combine); `doc_09__0` (подсчёт consecutive values) — только на **5-м** месте. Формально hit есть, но при `k=3` recursive уже потерял бы половину multi-hop (0.50).

---

## 5. Вывод

По **hit-rate@5** при hybrid-поиске стратегии **сравнялись (1.00)**, но **fixed** точнее ранжирует gold-документы (avg rank 1.17) при **в 7× меньшем** индексе; **recursive** лучше сохраняет границы Markdown-тредов и даёт более точные цитаты, но размывает multi-hop по рангам (Q6, Q11). Для этого корпуса разумный выбор — **recursive + hybrid (BM25)**: hit-rate не хуже, а фрагменты пригоднее для ответа; **fixed** предпочтителен, если критичны компактность индекса и top-1 precision.
