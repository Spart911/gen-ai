## 1. Новый инструмент `compare_periods`

**Сигнатура:**

```python
def compare_periods(
    metric: str,      # "key_rate" | "fx_USD" | "fx_EUR" | "fx_CNY" | "cpi" | "unemployment"
    period_a: str,    # "YYYY-MM" или "YYYY-MM-DD"
    period_b: str,
) -> dict:
    # → {"metric", "a": {"date", "value"}, "b": {"date", "value"},
    #     "delta", "ratio", "source"}
```

Внутри вызываются существующие `get_fx_rate`, `get_key_rate`, `get_inflation`, `get_unemployment` — логика не дублируется.

**Примеры запросов, где инструмент вызывается:**

| Запрос | Вызов | Результат |
|--------|-------|-----------|
| «Во сколько раз вырос курс USD с января 2022 по апрель 2026?» | `compare_periods(fx_USD, 2022-01, 2026-04)` | ratio = **0.96** (курс снизился) |
| «Во сколько раз вырос USD…» (eval Q5) | `compare_periods(fx_USD, 2022-01-01, 2026-04-01)` | ratio = **1.09** (курс вырос) |
| «На сколько п.п. изменилась ставка с марта 2022 по июнь 2024?» | `compare_periods(key_rate, 2022-03, 2024-06)` | delta = **−4 п.п.** |

## 3. Диагностика

### Ошибка 1: Инструмент вернул `error` (нет данных)

**Тип:** инструмент упал / отсутствуют данные в CSV

**Трасса** (run `e0b74d36`, Q3):

```json
{"step": 1, "call": "get_inflation", "args": {"year": 2026, "month": 5},
 "obs": {"error": "нет данных ИПЦ на 2026-05"}}
{"step": 2, "call": "get_inflation", "args": {"year": 2026, "month": 4},
 "obs": {"error": "нет данных ИПЦ на 2026-04"}}
{"step": 3, "call": "get_inflation", "args": {"year": 2026, "month": 3},
 "obs": {"cpi_yoy": 6.8, "source": "rosstat_csv"}}
```

**Причина:** в `cpi_ru_monthly.csv` нет апреля/мая 2026, а агент по умолчанию ищет «текущий» месяц (июнь → май → апрель).

**Что исправить:** в промпте указать «бери последний доступный месяц»; в `get_inflation` — fallback на ближайший месяц с данными (как у `get_fx_rate`).

---

### Ошибка 2: Арифметика без `calculate`

**Тип:** прочее — модель считает «в голове», минуя инструмент

**Трасса** (run `d8570bd9`, Q4):

```json
{"step": 1, "call": "get_key_rate", "args": {},
 "obs": {"rate": 16.0, "source": "fallback_csv"}}
{"step": 2, "final": "72 ÷ 16 = **4,5 года**"}
```

Аналогично Q3 (run `e0b74d36`): финал `16,0% − 6,8% = 9,2%` — `calculate` не вызывался.

**Причина:** правило «Do arithmetic ONLY via calculate» в промпте слабое; на простых операциях модель обходит калькулятор.

**Что исправить:** усилить промпт («любое +/−/×/÷ → обязательно `calculate`»); включить `--critic` для проверки, что числа в ответе есть в логе инструментов.

---

### Ошибка 3: Неоднозначная дата → противоположный ответ

**Тип:** неоднозначная интерпретация дат / схема периодов (один и тот же «январь 2022» → `2022-01` или `2022-01-01`)

**Трасса A** (ручная проверка, run `fe336804`):

```json
{"call": "compare_periods", "args": {"period_a": "2022-01", "period_b": "2026-04"},
 "obs": {"a": {"date": "2022-01-31", "value": 77.82}, "b": {"value": 74.88}, "ratio": 0.962}}
{"step": 2, "final": "курс **снизился** … в **0,96 раза**"}
```

**Трасса B** (eval Q5, run `f2a4b346`):

```json
{"call": "compare_periods", "args": {"period_a": "2022-01-01", "period_b": "2026-04-01"},
 "obs": {"a": {"date": "2022-01-01", "value": 74.29}, "b": {"value": 81.25}, "ratio": 1.094}}
{"step": 3, "final": "курс **вырос** … в **1,09 раза**"}
```

**Причина:** «январь 2022» можно передать как `2022-01` (конец месяца) или `2022-01-01` (начало). Курс сильно зависит от даты — ответ меняется на противоположный.

**Что исправить:** в схеме `compare_periods` явно писать: «если пользователь называет месяц — передавай `YYYY-MM`».

---

### Ошибка 4: Лишний вызов / неправильный порядок

**Тип:** неправильный порядок вызовов

**Трасса** (eval Q5, run `f2a4b346`):

```json
{"step": 1, "call": "compare_periods", "obs": {"ratio": 1.093654, ...}}
{"step": 2, "call": "calculate", "args": {"expression": "81.2504 / 74.2926"},
 "obs": {"result": 1.093654}}
```

`compare_periods` уже вернул готовый `ratio`, но агент повторно делит те же числа через `calculate`.

**Причина:** в описании `compare_periods` не сказано, что `ratio` можно сразу отдавать в ответ; модель не доверяет полю и перепроверяет.

**Что исправить:** в `schemas.py` добавить: «для вопросов “во сколько раз” используй `ratio` напрямую, `calculate` не нужен».

---

### Ошибка 5: Смешение периодов в составной метрике

**Тип:** прочее — методологическая ошибка

**Трасса** (Q10, run `1b539ed0`):

```json
{"step": 2, "call": "get_unemployment", "args": {"year": 2026, "month": 4},
 "obs": {"unemployment": 2.4}}
{"step": 3, "call": "get_inflation", "args": {"year": 2026, "month": 3},
 "obs": {"cpi_yoy": 6.8}}
{"step": 4, "call": "calculate", "args": {"expression": "6.8 + 2.4"}}
{"step": 5, "final": "индекс нищеты 9,2% (инфляция март + безработица апрель)"}
```

Индекс нищеты собран из **разных месяцев**: ИПЦ за март, безработица за апрель — потому что за апрель нет ИПЦ, а мартовская безработица не запрашивалась.

**Причина:** нет правила «составные показатели считай за один и тот же месяц»; агент берёт последнее доступное по каждой метрике отдельно.

**Что исправить:** в промпт: «misery index = inflation + unemployment за **один** месяц»; при отсутствии данных — явно оговаривать в ответе.

## 4. Вывод

**Самое хрупкое место системы — дисциплина вызова инструментов и интерпретация дат.**

Агент в целом получает правильные числа из ЦБ/CSV, но:

1. Считает простую арифметику без `calculate` (ломает eval и аудит).
2. По-разному понимает «январь 2022» → разные даты → противоположные выводы на одном вопросе.
3. Дублирует вызовы (`calculate` после `compare_periods`) и смешивает месяцы в составных метриках.
4. При отсутствии свежих данных ходит по месяцам вручную (3–5 шагов вместо одного).

Надёжнее всего усиливать **промпт + схемы инструментов** и добавить **ревизора (`--critic`)**, чем полагаться на «умность» модели в арифметике.
